# Agent Git Workflow

**Layer:** cross-cutting
**Status:** agreed — applies to all task execution by AI agents

---

## The Problem

Multiple AI coding agents working concurrently on different tasks will push commits to
the same repository. Without a branching strategy, agents overwrite each other's work,
merge conflicts accumulate silently, and main/develop becomes unstable.

## Convention

**One branch per task. One PR per branch.**

### Branch naming

```
agent/STORY-{NNN}/TASK-{NNN}-{short-slug}
```

Examples:
```
agent/STORY-001/TASK-001-leiden-algorithm
agent/STORY-001/TASK-002-leiden-summaries
agent/STORY-002/TASK-005-bitemporal-models
agent/STORY-005/TASK-009-same-as-embeddings
```

Every task file in `wiki/6_work/tasks/` has a `branch:` field in its frontmatter
specifying the exact branch name the assigned agent must use.

### Rules

1. **Never push directly to `develop` or `main`.** All work goes to a task branch.
2. **Branch from `develop`** (not from another task branch) unless the task's
   `blocked_by` list names another task that has not yet merged. In that case, branch
   from the blocking task's branch and rebase onto `develop` when it merges.
3. **One agent, one branch** at a time. An agent does not commit to a branch it did not create.
4. **Open a PR when `in-review`**: Task status `in-review` means a PR exists against
   `develop`. PR title = task ID + task title. PR description links to the task file.
5. **QA or Security Architect sign-off before merge.** The QA task for each story
   (`TASK-004`, `TASK-008`, `TASK-012`, ...) is the merge gate for the story's PRs.
6. **One commit per concern.** Each commit covers exactly one logical change. Examples of correct granularity:
   - `add event_time and ingestion_time properties to entity model`
   - `update LLM extraction prompt to request temporal bounds`
   - `add one-off migration task for existing nodes`
   These are three commits, not one. If a commit message needs "and", it is probably two commits.

7. **No AI attribution in commit messages.** Do not add `Co-Authored-By: Claude`, `Generated by:`, or any similar AI/tool signature to commits. Commit messages contain only what changed and why. The git history must read as if written by the human team.

### Dependency handling

If TASK-002 is `blocked_by: [TASK-001]`:
- TASK-001 agent works on `agent/STORY-001/TASK-001-leiden-algorithm`
- When TASK-001 is merged to `develop`, TASK-002 agent branches from `develop`
- If TASK-001 is not yet merged, TASK-002 agent branches from TASK-001's branch,
  notes it in the task log, and rebases once TASK-001 merges

### PR checklist (agent-generated PRs must include)

```markdown
## Task
[TASK-XXX](link to task file) — Short title

## What changed
- File 1: what was changed and why
- File 2: ...

## Definition of Done verified
- [ ] Primary deliverable complete
- [ ] Tests written and passing
- [ ] No regressions in existing test suite

## Branch
`agent/STORY-XXX/TASK-XXX-slug` → `develop`
```

---

## Why One Branch Per Task (not one per story)

A story may have 3-4 tasks assigned to different agents. If all agents push to one
story branch, they collide on every commit. Task-level branches isolate each agent's
work, keep PRs small and reviewable, and allow QA to review and merge incrementally.

---

## How This Is Enforced

The `branch:` field in each task's YAML frontmatter is the canonical branch name.
Agents must read their task file before starting work and use the specified branch.
Reza reviews PRs before merge; the QA engineer's sign-off on the test task is required
before the story is marked `done`.
